8// # Implementation notes
9// Several macros are defined hidden, suffixed with `·`, an publicly re-exported unsuffixed.
10// This makes them able to be imported from the root. E.g.: capture_last!, enumset!, set!…
11// See: https://github.com/rust-lang/rust/pull/52234#issuecomment-976702997
12//
13// # Warning regarding macro expansion
14// Some attributes (#[doc = …], #[rustfmt::skip], and other tooling-related ones) cause the
15// compiler to inspect or expand tokens earlier than normal, before the crate's macro resolution
16// graph is fully fixed. At that point, macros that are only introduced indirectly (for example
17// via helper macros, exported macros, or re-exports) may not yet be visible, even if they would
18// exist after full expansion. Because macro_rules! resolution depends on phase ordering, this
19// can lead to "resolution is stuck" errors where the compiler cannot prove which macro applies.
20// Defining the macro directly in the crate root avoids this, because it is visible in all phases.
21//
22// # Documentation for declarative macros
23// - [The Little Book of Rust Macros](https://veykril.github.io/tlborm/decl-macros.html)
24// - [Macros By Example](https://doc.rust-lang.org/reference/macros-by-example.html)
25// - [Specification](https://doc.rust-lang.org/reference/macro-ambiguity.html)
